# AI 代码仓退化识别 Agent 系统 — 需求文档

## 1. 项目概述

**项目名称**: 代码仓退化识别 Agent 系统
**项目定位**: 基于 LLM 的智能 Agent 系统，能够自主感知、推理、诊断代码仓库的质量退化问题。Agent 依据数据主动发现问题、追根溯源、交叉验证，输出有深度的诊断报告
**目标用户**: 需要监控代码质量趋势的开发团队和技术管理者
**数据来源**: 代码质量展示平台（CodeQualityShow）REST API，降级时使用 Git 仓库分析

---

## 2. 核心概念

### 2.1 退化定义

退化指代码质量随时间推移而恶化的趋势，具体表现为：

| 维度           | 说明                         | 示例                     |
| -------------- | ---------------------------- | ------------------------ |
| 代码重复率上升 | 重复代码块增多               | 从 8% 增长到 18%         |
| 圈复杂度增高   | 高复杂度函数/类增加          | 新增 15 个高圈复杂度函数 |
| 设计问题累积   | 循环依赖、紧耦合、上帝类增加 | 循环依赖从 3 增至 12     |
| 坏味道蔓延     | 数据泥团、消息链等反模式增多 | 数据泥团新增 8 处        |
| 技术债增长     | 综合技术债指标上升           | debtTotal 持续攀升       |
| 代码量异常增长 | 短期内代码行数暴增           | 未重构的快速堆叠         |
| 巨型文件增多   | 超过大小阈值的文件增加       | 新增 5 个巨型文件        |

### 2.2 Agent 分析模式

Agent 的核心不是固定流水线，而是**自主推理循环**。LLM 依据当前发现的问题，自主决定下一步调用什么工具、是否需要深入调查。以下为典型推理路径（非唯一路径）：

```
用户: "分析一下支付域最近的代码质量情况"
  │
  ▼
Agent 推理: 需要先了解有哪些域和可用数据
  → 调用 list_domains → 发现"支付域"存在
  → 调用 list_available_months → 发现有 202601-202606 数据
  │
  ▼
Agent 推理: 先看整体对比趋势
  → 调用 compare_snapshots(支付域, 202601, 202606)
  → 发现: 代码重复率 ↑10%, 循环依赖 ↑9, 技术债 ↑89
  │
  ▼
Agent 推理: 循环依赖涨幅最大，需下钻定位
  → 调用 get_domain_detail(支付域, 202606, dimension=group)
  → 发现: "支付核心组"循环依赖 8 个，占总量的 67%
  │
  ▼
Agent 推理: 再深入看微服务级别
  → 调用 get_domain_detail(支付域, 202606, dimension=microservice)
  → 发现: "PaymentService" 有 5 个循环依赖 + 上帝类 3 个
  │
  ▼
Agent 推理: 交叉验证基线数据
  → 调用 analyze_repo(PaymentService, 202601)
  → 发现: 6 个月前循环依赖仅 1 个，上帝类 0 个
  │
  ▼
Agent 推理: 问题确认，输出结构化 DegradationReport
```

## 3. 功能需求

### 3.1 核心功能

| 功能          | 描述                                                 | 优先级 |
| ------------- | ---------------------------------------------------- | ------ |
| 质量快照获取  | 从质量平台获取指定月份/微服务的代码质量数据          | P0     |
| 月度对比分析  | 对比两个月份的质量快照，计算各指标变化趋势           | P0     |
| 退化报告生成  | Agent 综合分析数据，自主推理后生成结构化退化分析报告 | P0     |
| 域/微服务下钻 | 按域、分组、微服务维度逐层下钻分析                   | P0     |
| 自由追问      | 对已有报告进行追问（如"详细说说循环依赖问题"）       | P0     |
| 报告持久化    | 保存分析报告，支持历史查询                           | P1     |

### 3.2 扩展功能

| 功能         | 描述                                        | 优先级 |
| ------------ | ------------------------------------------- | ------ |
| Git 降级分析 | 质量平台不可用时，基于 git 命令进行基础分析 | P1     |
| 定时巡检     | 定期自动分析，发现退化及时告警              | P2     |
| 报告导出     | 将报告导出为 Markdown/HTML 文件             | P2     |

## 4. 数据来源

### 4.1 代码质量展示平台

- **前端**: http://localhost:5174（Vue 3 + Element Plus + ECharts）
- **后端 API**: http://localhost:31500（Spring Boot 3.5.0）
- **数据库**: MySQL 9.5.0

### 4.2 可用的 API 端点

| API 端点                        | 方法 | 用途               | 关键参数                             |
| ------------------------------- | ---- | ------------------ | ------------------------------------ |
| `/show/code-info`               | GET  | 代码质量数据       | `month`(YYYYMM), `baseMonth`(YYYYMM) |
| `/show/code-review`             | GET  | 代码审查统计       | `month`(YYYYMM)                      |
| `/show/code-info/history`       | GET  | 12 个月历史趋势    | `month`, `domain`                    |
| `/show/domain/info`             | GET  | 域/分组/微服务列表 | -                                    |
| `/show/domain/detail`           | GET  | 按维度下钻详情     | `month`, `domain`, `dimension`       |
| `/show/code-info/microservices` | GET  | 微服务级数据       | `month`                              |

### 4.3 对比查询

`/show/code-info` 支持 `baseMonth` 参数，一次请求获取两个月对比数据。返回数据中每个指标同时包含当前值和上一期值（如 `codeNumber` / `codeNumberPre`）。

### 4.4 质量指标

平台提供 50+ 指标，退化分析精选核心指标如下：

| 类别     | 字段                                                         | 中文名                                                       |
| -------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| 重复     | `codeLineDuplicationRate`, `duplicateLines`, `duplicateFiles`, `internalRepetition`, `externalDuplication` | 代码重复率、重复行数、重复文件数、内部重复、外部重复         |
| 复杂度   | `largeCyclomaticComplexity`, `largeFunctionNumbers`, `complexClasses`, `complexModules` | 高圈复杂度函数数、超大函数数、复杂类数、复杂模块数           |
| 规模     | `codeNumber`, `fileNumber`, `hubCount`, `hugeFiles`          | 代码总行数、文件数、仓库数、巨型文件数                       |
| 设计     | `circularDependencies`, `tightCoupling`, `godClasses`, `godModule`, `debtTotal`, `unnecessaryCoupling` | 循环依赖数、紧耦合数、上帝类数、上帝模块数、技术债总量、不必要耦合 |
| 坏味道   | `dataMires`, `dataClasses`, `messageChain`, `redundantCodeBlocks`, `attachmentPattern`, `schizophreniaClasses` | 数据泥团、数据类、消息链、冗余代码块、依恋情节、精神分裂类   |
| 代码审查 | `reviewNumber`, 严重级别分布, 修复状态                       | 审查数量、各级别数量、修复状态                               |

### 4.5 维度下钻

平台支持三级组织维度：域 (domain) → 分组 (group/subgroupName) → 微服务 (microservice)。通过 `dimension` 参数切换。

### 4.6 降级方案

质量平台不可用时，通过 GitDataProvider 基于 git 命令进行基础分析（仅覆盖 total_lines, total_files, duplication_rate 等基础指标）。

## 5. 用户交互

### 5.1 CLI 命令

在现有 REPL 基础上新增：

| 命令      | 说明                          |
| --------- | ----------------------------- |
| `analyze` | 对指定仓库/微服务执行退化分析 |
| `reports` | 列出已保存的退化分析报告      |

### 5.2 分析模式交互

```
【用户】: analyze
请输入微服务/仓库名称 (留空分析全部): MyService
请输入对比起始月份 (YYYYMM，留空自动选择): 202601
请输入对比结束月份 (YYYYMM，默认最近月份):
正在分析...
  [工具调用] list_domains({})
  [工具结果] ...
  [工具调用] compare_snapshots(...)
  [工具结果] ...
  [工具调用] get_domain_detail(...)
  [工具结果] ...
  ...（Agent 自主决定调用路径）

================================================
  代码仓库退化分析报告
================================================
仓库/微服务: MyService
对比: 202601 → 202606
综合健康度: 62/100 (趋势: 退化)
------------------------------------------------
核心指标: ...
风险区域: ...
改进建议: ...
================================================

报告已保存，ID: a1b2c3d4
```

### 5.3 追问模式交互

```
用户: 分析一下支付域
Agent: [自主分析，输出报告]

用户: PaymentService 的循环依赖具体是哪些模块之间？
Agent: 让我下钻看看 → 调用 get_domain_detail → 返回下钻结果及分析

用户: 这两个组为什么会互相依赖？
Agent: 从数据看，支付核心组有 3 个上帝类，它们同时引用了风控组的接口...
```

## 6. 技术约束

| 约束       | 说明                                                         |
| ---------- | ------------------------------------------------------------ |
| 平台       | Windows，需兼容 Windows 路径和 git 命令                      |
| LLM        | 支持多种模型（MiniMax-M2.7、GLM5.1 等），需兼容不同模型的上下文窗口和 function calling 能力差异 |
| 上下文窗口 | 有限（4K-8K token），需主动管理上下文                        |
| 异步       | MCP SDK 异步，需通过 async_utils.py 桥接                     |
| 依赖       | 复用项目已有的 httpx 依赖，不引入新重型依赖                  |
| 向后兼容   | 不修改现有模块的公共接口，原有对话功能不受影响               |
| API 基址   | 通过 `QUALITY_API_BASE` 环境变量配置，默认 `http://localhost:31500` |

---

## 7. 非功能需求

| 需求类型   | 描述                                          |
| ---------- | --------------------------------------------- |
| 性能       | 单次快照获取 < 5 秒；完整分析报告生成 < 60 秒 |
| 可靠性     | 快照缓存机制，相同 repo+month 不重复请求 API  |
| 可扩展性   | 数据源通过协议抽象，平台不可用时降级到 Git    |
| 容错       | LLM 输出解析失败时退化处理，不中断流程        |
| 上下文安全 | 上下文窗口溢出时自动压缩，用户消息不可丢弃    |
| 易用性     | CLI 交互引导，自动选择默认月份对比            |
