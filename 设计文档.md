## 4. 数据模型

### 4.1 QualitySnapshot

```python
@dataclass
class QualitySnapshot:
    # 标识
    repo_path: str              # 仓库/微服务名称
    ref: str                    # 月份标识 (YYYYMM)
    timestamp: float

    # 重复类
    duplication_rate: float     # codeLineDuplicationRate
    duplicate_lines: int        # duplicateLines
    duplicate_files: int        # duplicateFiles

    # 复杂度类
    high_complexity_count: int  # largeCyclomaticComplexity
    large_functions: int        # largeFunctionNumbers
    complex_classes: int        # complexClasses

    # 规模类
    total_lines: int            # codeNumber
    total_files: int            # fileNumber
    huge_files: int             # hugeFiles

    # 设计类
    circular_dependencies: int  # circularDependencies
    tight_coupling: int         # tightCoupling
    god_classes: int            # godClasses
    debt_total: float           # debtTotal

    # 坏味道类
    data_mires: int             # dataMires
    message_chain: int          # messageChain
    redundant_code_blocks: int  # redundantCodeBlocks
```

### 4.2 DiffSummary

```python
@dataclass
class DiffSummary:
    ref_from: str               # 起始月份 (YYYYMM)
    ref_to: str                 # 结束月份 (YYYYMM)
    snapshot_from: QualitySnapshot
    snapshot_to: QualitySnapshot
```

### 4.3 MetricValue

```python
@dataclass
class MetricValue:
    name: str           # 指标中文名
    value: float        # 当前值
    previous_value: float  # 上期值
    change: float       # 变化量
    unit: str           # 单位
    trend: str          # "improving" | "stable" | "degrading"
    severity: str       # "info" | "warning" | "critical"
```

### 4.4 RiskArea

```python
@dataclass
class RiskArea:
    path: str               # 域/分组/微服务名称
    description: str        # 风险描述（中文）
    metrics: List[MetricValue]
    recommendation: str     # 改进建议（中文）
```

### 4.5 CausalChain

```python
@dataclass
class CausalChain:
    root: str               # 根因描述
    chain: List[str]        # 传导路径，如 ["循环依赖→紧耦合", "紧耦合→上帝类"]
    evidence: str           # 数据支撑说明
```

### 4.6 DegradationReport

```python
@dataclass
class DegradationReport:
    id: str                     # 报告唯一 ID（截断 UUID）
    repo_path: str              # 分析的仓库/微服务
    ref_from: str               # 对比起始月份 (YYYYMM)
    ref_to: str                 # 对比结束月份 (YYYYMM)
    created_at: float           # 创建时间
    overall_score: float        # 综合健康度 (0-100)
    overall_trend: str          # "improving" | "stable" | "degrading"
    metrics: List[MetricValue]  # 核心指标列表
    risk_areas: List[RiskArea]  # 风险区域列表
    causal_chains: List[CausalChain]  # 因果链列表
    summary: str                # 中文总结
    recommendations: List[str]  # 中文改进建议
```

## 5. 指标体系

### 5.1 指标定义

| 指标 key                | 中文名         | 单位 | 方向     | 数据来源字段                |
| ----------------------- | -------------- | ---- | -------- | --------------------------- |
| `duplication_rate`      | 代码重复率     | %    | 越低越好 | `codeLineDuplicationRate`   |
| `duplicate_lines`       | 重复代码行数   | 行   | 越低越好 | `duplicateLines`            |
| `high_complexity_count` | 高圈复杂度函数 | 个   | 越低越好 | `largeCyclomaticComplexity` |
| `complex_classes`       | 复杂类数       | 个   | 越低越好 | `complexClasses`            |
| `total_lines`           | 代码总行数     | 行   | 中性     | `codeNumber`                |
| `total_files`           | 文件数         | 个   | 中性     | `fileNumber`                |
| `huge_files`            | 巨型文件数     | 个   | 越低越好 | `hugeFiles`                 |
| `circular_dependencies` | 循环依赖       | 个   | 越低越好 | `circularDependencies`      |
| `tight_coupling`        | 紧耦合         | 个   | 越低越好 | `tightCoupling`             |
| `god_classes`           | 上帝类         | 个   | 越低越好 | `godClasses`                |
| `debt_total`            | 技术债总量     | -    | 越低越好 | `debtTotal`                 |
| `data_mires`            | 数据泥团       | 个   | 越低越好 | `dataMires`                 |
| `message_chain`         | 消息链         | 个   | 越低越好 | `messageChain`              |
| `redundant_code_blocks` | 冗余代码块     | 个   | 越低越好 | `redundantCodeBlocks`       |

### 5.2 趋势判定规则

- **improving**: 指标向好方向变化超过 5%（或绝对值变化 > 阈值）
- **stable**: 变化幅度在 ±5% 以内
- **degrading**: 指标向坏方向变化超过 5%（或绝对值变化 > 阈值）

### 5.3 严重度判定规则

| 严重度       | 条件                                                         |
| ------------ | ------------------------------------------------------------ |
| **critical** | 指标方向为"越低越好"且趋势为 degrading，且当前值超过严重阈值 |
| **warning**  | 指标趋势为 degrading 但当前值未超阈值；或趋势为 stable 但当前值超过警告阈值 |
| **info**     | 趋势为 improving 或 stable，且当前值在正常范围               |

注意：趋势为 stable 但当前值已超过警告阈值的指标，仍标记为 warning。此类指标虽未恶化，但已处于不健康状态。

## 6. MCP 工具设计

### 6.1 工具列表

| 工具名                     | 描述                   | 输入参数                                                     | 输出                  |
| -------------------------- | ---------------------- | ------------------------------------------------------------ | --------------------- |
| `analyze_repo`             | 获取某月份的质量快照   | `repo_path`(微服务名,必填), `month`(YYYYMM,可选默认最近)     | 精简后的质量指标 JSON |
| `compare_snapshots`        | 对比两个月份的质量变化 | `repo_path`(必填), `month_from`(YYYYMM,必填), `month_to`(YYYYMM,必填) | 精简后的对比指标 JSON |
| `list_available_months`    | 列出有数据的月份       | `repo_path`(可选), `count`(可选,默认 12)                     | 月份列表              |
| `list_domains`             | 列出域/分组/微服务     | 无                                                           | 组织层级结构          |
| `get_domain_detail`        | 按维度下钻详情         | `domain`(必填), `month`(YYYYMM,必填), `dimension`(group/subgroup/microservice,必填) | 下钻数据列表          |
| `list_degradation_reports` | 列出已保存的报告       | `repo_path`(可选)                                            | 报告摘要列表          |

### 6.2 工具结果精简

MCP Server 返回工具结果时，对原始平台 API 响应进行结构化精简，按三级分类输出：

| 级别                | 筛选条件                               | 输出形式                                                   | 目的                           |
| ------------------- | -------------------------------------- | ---------------------------------------------------------- | ------------------------------ |
| **high_attention**  | 趋势为 degrading，或当前值超过严重阈值 | 完整字段（name, value, previous, change, trend, severity） | Agent 优先处理的风险项         |
| **needs_attention** | 趋势为 stable 但当前值已超过警告阈值   | 精简字段（name, value, severity）                          | 指标虽未恶化但已处于不健康状态 |
| **overview**        | 趋势为 stable 且值正常                 | 仅统计数量（stable_count, improving_count）                | 全局视角                       |

**精简示例：**

```
原始 API 返回（约 2000 token）:
  { "codeNumber": 52300, "codeNumberPre": 39500, "codeLineDuplicationRate": 18.5,
    "circularDependencies": 12, "hugeFiles": 7, "hugeFilesPre": 6, ... 共 50+ 字段 }

MCP 工具精简后返回（约 400 token）:
  { "month": "202606", "baseMonth": "202601",
    "high_attention": [
      {"name": "代码重复率", "value": 18.5, "previous": 8.2, "change": "+10.3%",
       "trend": "degrading", "severity": "critical"},
      {"name": "循环依赖", "value": 12, "previous": 3, "change": "+9",
       "trend": "degrading", "severity": "critical"}
    ],
    "needs_attention": [
      {"name": "巨型文件", "value": 7, "severity": "warning"}
    ],
    "overview": {"stable_count": 6, "improving_count": 1}
  }
```

## 7. Agent 核心设计

### 7.1 System Prompt

System Prompt 仅包含角色定义和输出约束，推理策略由 Skill 按阶段注入：

```
你是一位资深代码架构师，专门负责代码仓库退化问题的诊断和分析。

分析风格:
- 数据驱动：每个结论必须有指标数据支撑
- 追根溯源：不只看表象，要找到退化的根因和传导链
- 重点突出：聚焦最严重的问题，不面面俱到
- 可操作：建议要具体到"做什么"，不只是"应该改善"

输出约束:
当你完成分析后，输出结构化 JSON 报告:
{ "overall_score": <0-100>, "overall_trend": "...",
  "metrics": [...], "risk_areas": [...], "causal_chains": [...],
  "summary": "...", "recommendations": [...] }
```

### 7.2 Skill 系统

#### Skill 定义

```python
class Skill:
    name: str               # 技能名称
    description: str        # 技能描述
    prompt_template: str    # 注入到上下文中的提示词片段
    tools_required: List[str]  # 该技能需要的 MCP 工具
```

#### 预定义 Skill

| Skill               | 触发场景                 | 注入的提示词                                                 | 使用的工具                                               |
| ------------------- | ------------------------ | ------------------------------------------------------------ | -------------------------------------------------------- |
| `scope_survey`      | 分析开始，需了解全局     | "先了解组织结构和时间范围，不急于下钻。列出域和月份后，选择最值得分析的范围。" | `list_domains`, `list_available_months`                  |
| `anomaly_detect`    | 需快速定位退化指标       | "对比两个时间点，找出退化最严重的 3-5 个指标。重点关注'越低越好'但趋势为 degrading 的指标。注意指标间的关联。" | `compare_snapshots`, `analyze_repo`                      |
| `deep_dive`         | 发现异常指标，需下钻定位 | "对异常指标下钻到分组/微服务级别，定位问题集中的团队或服务。对比同一维度下的多个指标，判断是单一问题还是系统性退化。" | `get_domain_detail`                                      |
| `cross_validate`    | 需验证结论或检查影响范围 | "交叉验证你的发现：检查相邻/依赖的微服务是否受影响；对比历史趋势判断是突发还是累积；若数据不支持结论，重新调查。" | `analyze_repo`, `compare_snapshots`, `get_domain_detail` |
| `synthesize_report` | 信息充分，准备输出报告   | "综合所有发现，识别因果链和根因，输出结构化 JSON 报告。确保每个结论有数据支撑，建议要具体可操作。" | 无                                                       |

#### Skill 调用机制

Agent 在 ReAct 循环中依据当前分析阶段自动匹配 Skill，将对应 Skill 的 prompt_template 注入下一轮 system 消息。阶段转换逻辑：

```
ReAct 第 1 轮: 用户要求分析 → 匹配 scope_survey → 注入对应提示词
ReAct 第 3 轮: 范围已确定 → 匹配 anomaly_detect → 替换 Skill 提示词
ReAct 第 5 轮: 发现异常指标 → 匹配 deep_dive → 替换 Skill 提示词
ReAct 第 8 轮: 信息充分 → 匹配 synthesize_report → 替换 Skill 提示词
```

#### Skill 组合预设

| 场景     | Skill 组合                                                   | 预估轮数 |
| -------- | ------------------------------------------------------------ | -------- |
| 快速扫描 | scope_survey → anomaly_detect → synthesize_report            | 3-5 轮   |
| 标准分析 | scope_survey → anomaly_detect → deep_dive → synthesize_report | 5-8 轮   |
| 深度诊断 | scope_survey → anomaly_detect → deep_dive → cross_validate → synthesize_report | 8-15 轮  |

默认使用标准分析，发现严重退化时自动升级为深度诊断。

### 7.3 ReAct 循环

```
┌─────────────────────────────────────────────┐
│              ReAct 循环                       │
│                                              │
│  1. 思考 (Reasoning)                         │
│     LLM 分析当前已知信息，决定下一步          │
│                                              │
│  2. 行动 (Acting)                            │
│     LLM 选择工具 + 参数，发起 tool_calls     │
│                                              │
│  3. 观察 (Observation)                       │
│     工具返回结果，加入对话上下文              │
│                                              │
│  4. 重复 或 收敛                             │
│     - 发现新问题 → 继续循环                   │
│     - 信息充分 → 输出最终报告                 │
│     - 达到最大迭代 → 输出当前结论              │
└─────────────────────────────────────────────┘
```

参数：

| 参数                  | 默认值 | 说明                              |
| --------------------- | ------ | --------------------------------- |
| `max_tool_iterations` | 15     | 单次分析最大工具调用轮数          |
| `converge_check`      | 自动   | LLM 连续 2 轮不调用工具时自动收敛 |

### 7.4 Agent 接口

```python
class DegradationAgent:
    def __init__(
        self,
        storage: Optional[DegradationStorage] = None,
        mcp_client = None,
        max_tool_iterations: int = 15,
        system_prompt: Optional[str] = None,
    ): ...

    def analyze(
        self,
        repo_path: str = None,
        month_from: str = None,
        month_to: str = None,
    ) -> DegradationReport:
        """执行完整的退化分析，Agent 自主决定分析路径，返回结构化报告"""

    def chat(self, message: str) -> str:
        """基于已有分析上下文自由对话，Agent 保留分析记忆"""
```

### 7.5 Agent 能力矩阵

| 场景       | Agent 行为                        | 推理示例                                                     |
| ---------- | --------------------------------- | ------------------------------------------------------------ |
| 异常检测   | 对比后聚焦退化最严重的 3-5 个指标 | "代码重复率涨了 10%，但文件数只涨了 15%，非规模增长导致的合理重复" |
| 根因定位   | 发现异常后主动下钻到分组/微服务   | "PaymentService 循环依赖从 1 涨到 5，同组其他服务正常，问题出在它自身" |
| 因果推断   | 多指标同时退化时识别因果链        | "循环依赖→强制紧耦合→无法拆分→演变为上帝类，根因是循环依赖"  |
| 影响评估   | 核心服务退化时检查依赖方          | "UserService 紧耦合严重，PaymentService 依赖它，可能被传导"  |
| 优先级排序 | 多指标退化时判断处理优先级        | "技术债总量最高但增长缓慢，循环依赖增速最快，后者更紧迫"     |
| 历史对比   | 回查更早数据判断长期趋势          | "对比 12 个月趋势，循环依赖从 202603 开始加速增长，非偶发"   |
| 追问深挖   | 用户追问后调用新工具下钻          | "让我看下具体是哪个分组的重复率最高" → 调用 get_domain_detail |

### 7.6 记忆机制

| 记忆类型     | 存储                         | 生命周期 | 用途                           |
| ------------ | ---------------------------- | -------- | ------------------------------ |
| 工具调用历史 | Conversation.messages        | 单次分析 | 避免重复调用已查过的工具       |
| 分析结论     | DegradationReport            | 持久化   | 追问时引用之前的结论           |
| 快照缓存     | DegradationStorage.snapshots | 跨会话   | 相同 repo+month 不重复请求 API |
| 历史报告     | DegradationStorage.reports   | 跨会话   | 支持跨次对比                   |

## 8. 上下文管理

### 8.1 问题定义

MiniMax-M2.7 上下文窗口有限（4K-8K token），GLM5.1 等其他模型的窗口大小各异。系统需适配不同模型的上下文容量，一次深度分析可能产生 10,000-25,000 token 的上下文消耗。主要膨胀源为工具返回结果（300-2000 token/轮）。

### 8.2 三层管理策略

#### 策略一：MCP Server 端结果精简

详见 6.2 节。通过三级分类输出，将单次工具结果从约 2000 token 压缩到约 400 token。

#### 策略二：Agent 端分层压缩

当对话历史超过 token 预算阈值时，ContextManager 对历史消息执行分层压缩。

**核心原则：用户消息不可丢弃。**

用户消息包含分析意图、关注方向、偏好倾向和纠正指令，任何压缩操作均须完整保留所有用户消息。

**消息优先级：**

| 优先级         | 消息类型                        | 压缩策略         | 保留理由                       |
| -------------- | ------------------------------- | ---------------- | ------------------------------ |
| P0（不可压缩） | 用户消息                        | 完整保留         | 包含用户意图、偏好和纠正指令   |
| P0（不可压缩） | system prompt                   | 完整保留         | 角色定义和当前 Skill           |
| P1（近期保留） | 最近 K 轮 assistant + tool 消息 | 完整保留         | 当前推理的直接上下文           |
| P2（可压缩）   | 超过 K 轮的 tool 返回结果       | 压缩为结构化摘要 | 信息密度低，摘要可保留关键结论 |
| P2（可压缩）   | 超过 K 轮的 assistant 推理      | 压缩为结论陈述   | 保留结论，丢弃过程             |

**压缩流程：**

```
每轮 ReAct 结束后触发检查
         │
         ▼
   token 估算 > 阈值？
         │
      否 │         是
         │          │
         │          ▼
         │    执行分层压缩：
         │    1. 提取全部用户消息 → 完整保留
         │    2. 提取最近 K 轮消息 → 完整保留
         │    3. 提取早期 tool 结果 → 生成结构化摘要
         │    4. 提取早期 assistant 推理 → 提取结论陈述
         │    5. 按 [system][用户消息][摘要][近期消息] 重组
         │          │
         └──────────┘
              │
              ▼
        下一轮 ReAct 循环
```

**压缩示例：**

压缩前（6 轮工具调用，约 12,000 token）：

```
[system] 你是代码架构师...
[user] 分析支付域，重点关注循环依赖问题
[assistant] 先了解范围 → tool_calls: [list_domains, list_available_months]
```

```python
class ContextManager:
```

#### 策略三：Skill 提示词动态替换

System prompt 仅包含角色定义和输出约束（约 300 token），推理策略由当前 Skill 按阶段动态注入（约 200 token/阶段），而非一次性加载全部策略文本。

### 8.3 上下文预算分配

| 内容                             | 预算占比 | 预估 token | 说明           |
| -------------------------------- | -------- | ---------- | -------------- |
| System Prompt（角色 + 输出约束） | ~5%      | 300        | 固定不变       |
| 当前 Skill 提示词                | ~3%      | 200        | 按阶段动态替换 |
| 用户消息（全部）                 | ~5%      | 200-500    | 不可压缩       |
| 分析摘要（压缩历史）             | ~12%     | 800-1000   | 结构化摘要     |
| 最近 K 轮完整消息                | ~40%     | 2500-3000  | K 值动态调整   |
| LLM 输出空间                     | ~35%     | 2000-2500  | 推理和报告生成 |

K 值（保留的近期完整轮数）根据当前上下文占用动态计算，非固定值。`MAX_CONTEXT_TOKENS` 依据当前使用的模型动态配置（如 MiniMax-M2.7 设为 6000，GLM5.1 等窗口更大的模型可适当上调），通过 `Config.max_context_tokens` 管理。

## 9. DataProvider 设计

### 9.1 协议定义

```python
class DataProvider(Protocol):
    def get_snapshot(self, repo_path: str, ref: str) -> QualitySnapshot: ...
    def list_refs(self, repo_path: str, count: int) -> List[str]: ...
    def get_diff_summary(self, repo_path: str, ref_from: str, ref_to: str) -> DiffSummary: ...
    def list_domains(self) -> List[Dict]: ...
    def get_domain_detail(self, domain: str, ref: str, dimension: str) -> List[Dict]: ...
```

### 9.2 QualityPlatformProvider（首选）

| 方法                                          | API 调用                                                     |
| --------------------------------------------- | ------------------------------------------------------------ |
| `get_snapshot(repo, month)`                   | GET `/show/code-info?month={month}`                          |
| `list_refs(repo, count)`                      | GET `/show/code-info/history`                                |
| `get_diff_summary(repo, from, to)`            | GET `/show/code-info?month={to}&baseMonth={from}`            |
| `list_domains()`                              | GET `/show/domain/info`                                      |
| `get_domain_detail(domain, month, dimension)` | GET `/show/domain/detail?month={month}&domain={domain}&dimension={dimension}` |

配置：`QUALITY_API_BASE` 环境变量，默认 `http://localhost:31500`。使用 `httpx` 发送 HTTP 请求。

### 9.3 GitDataProvider（降级）

基于 git 命令 + 简单文本分析，仅覆盖 QualitySnapshot 中的基础字段（total_lines, total_files, duplication_rate），其余为 0。适用于离线场景或质量平台不可用时。

## 10. DegradationStorage 设计

### 10.1 存储结构

```
data/degradation/
  snapshots/{repo_hash}_{ref_hash}.json    # 质量快照缓存
  reports/{report_id}.json                  # 分析报告
  reports/index.json                        # 报告索引
```

### 10.2 接口

```python
class DegradationStorage:
    # 快照（带缓存，相同 repo+month 不重复请求）
    def save_snapshot(self, snapshot: QualitySnapshot) -> str
    def load_snapshot(self, repo_path: str, month: str) -> Optional[QualitySnapshot]

    # 报告
    def save_report(self, report: DegradationReport) -> str
    def load_report(self, report_id: str) -> Optional[DegradationReport]
    def list_reports(self, repo_path: str = None) -> List[Dict]
    def delete_report(self, report_id: str) -> bool
```

快照文件命名：`{repo_hash}_{ref_hash}.json`（SHA-256 截断至 12 字符）。报告 ID 采用截断 UUID（同现有 Conversation）。
